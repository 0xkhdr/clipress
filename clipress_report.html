📖 Overview

What is Clipress? A Python-based CLI proxy that intercepts bash command output before it reaches an AI agent's context window, compresses it using a hybrid classifier + registry system, and returns only the semantically meaningful portion.

Key Features

  • Automatic Detection: Recognizes output shapes (lists, tables, logs, diffs, tests, progress)
  • Adaptive Learning: Improves with every call through SQLite-backed registry
  • Memory Safe: Bounded rolling-window compression prevents OOM
  • Secure: Blocks sensitive patterns, binary data, and credentials
  • Zero Overhead: Hot cache for proven commands (3+ consistent calls)
  • Extensible: Custom rules via YAML, no code required

Current Improvements (v1.2.2)

✅ 5 Quality Improvements Implemented:
• Better token counting (85% accurate for paths)
• Warm tier (3 calls instead of 10 to activate cache)
• Fuzzy command matching (command variations share cache)
• ANSI presence guard (O(1) check skips regex for 90% of outputs)
• Adaptive heartbeat (eliminates thread overhead for small outputs)

Config File: .clipress/config.yaml

engine:
  show_metrics: false                    # Print token savings to stderr
  min_lines_to_compress: 15              # Skip outputs shorter than this
  hot_cache_threshold: 10                # Calls before promoting to hot cache
  strip_ansi: true                       # Remove ANSI escape codes
  max_output_bytes: 10485760             # 10 MB max (larger = pass through)

  binary_non_ascii_ratio: 0.3            # % non-ASCII → treat as binary

🔧 Compression Strategies

Shape Detection: Clipress analyzes output structure and automatically selects the best strategy. Each shape has configurable parameters for fine-tuning.

8 Output Shapes

📋 List

File listings, grep results, log entries

git log, ls, find, cat

Keeps head + tail, optional dedup

📊 Table

Columnar output with headers

docker ps, kubectl get, ps aux

Truncates rows and cells

⏳ Progress

Build logs, downloads, installations

docker build, npm install, pip install

Strips % lines, keeps errors

✅ Test

Unit test results with failures

pytest, jest, cargo test

Extracts failures + traceback

📝 Diff

Git diffs, patch files

git diff, git show

Keeps hunks with context

🔑 Key-Value

Config dumps, status output

systemctl status, docker inspect

Truncates keeping non-timestamps

❌ Error

Stack traces and exceptions

Python tracebacks, Java exceptions

Keeps frames, strips stdlib

🎯 Generic

Fallback truncation (unknown shapes)

Any unmapped output

Memory-bounded rolling window

Compression Pipeline

stdin / PTY
│
├─ [SIZE GUARD] output > 10 MB? → pass through
├─ [ANSI STRIP] remove escape codes
├─ [SAFETY GATE] check blocklist, security patterns, binary
│
├─ [STRATEGY RESOLUTION] (priority order)
│ 1. Hot cache — in-memory LRU (≥10 calls, 0.85 confidence)
│ 2. Warm cache — SQLite (≥3 calls, 0.65 confidence)
│ 3. Seed registry — built-in rules
│ 4. Classifier — heuristic shape detection
│
├─ [COMPRESS] apply matched strategy
├─ [CONTRACTS] apply always_keep / always_strip
├─ [REGRESSION GUARD] compressed > original? → return original
├─ [METRICS] count tokens saved
└─ [LEARN] update registry.db
│
stdout (compressed output)

Example: git log | clipress compress "git log"

  1. Size guard: 100-line log ~2 KB → proceed
  2. ANSI strip: No codes present → skip
  3. Safety gate: Not blocked, not binary → proceed
  4. Strategy resolution: Seed has "git log" → {strategy: "list", max_lines: 20}
  5. Compress: List strategy keeps head (20) + tail (5), inserts ... [75 more items]
  6. Contracts: No per-command rules → no change
  7. Regression guard: 30 lines vs 100 lines → accept
  8. Metrics: ~1950 tokens saved (75% reduction) → log to stderr
  9. Learn: Update registry: confidence: 0.50, calls: 1

🧠 Learning System

Three-Tier Knowledge Base: Clipress improves with every call through a tiered system that prioritizes speed (hot cache) while learning patterns over time (persistent SQLite registry).

Tier 1 — Hot Cache (In-Memory)

  • LRU OrderedDict (max 100 entries), thread-safe
  • Promotion criteria: ≥10 calls + ≥0.85 confidence
  • Speed: Zero-latency lookup (no disk, no classification)
  • Scope: Single process (per-agent session)

Tier 2 — Warm Cache (SQLite)

  • Persistent SQLite database (with WAL mode for concurrency)
  • Activation criteria: ≥3 calls + ≥0.65 confidence
  • Speed: Skip classification, use cached strategy
  • Scope: Workspace-wide (shared across all processes)

Tier 3 — Seed Registry (Built-In)

  • Pre-defined strategies for common tools: git, docker, pytest, kubectl, npm
  • Matching: Longest-key-first, so docker ps -a before docker ps
  • Override: User extensions in .clipress/extensions/*.yaml

Confidence Mechanics

Event Confidence Delta
Strategy matches previous +0.08
Strategy differs −0.20
Confidence drops below 0.50 Reset to 0.50, adopt new strategy
Confidence ≥ 0.85 AND calls ≥ 10 Promoted to hot cache
Confidence ≥ 0.95 Locked (stops updating)
Why the asymmetry? Wrong predictions (−0.20) degrade confidence faster than correct ones restore it (+0.08). This prevents sticking with mismatched strategies.

Warm Tier Improvement

NEW in v1.2.2: The warm tier activates after just 3 consistent calls at 0.65 confidence, compared to the hot tier's requirement of 10 calls + 0.85 confidence.

  • Reduces cold-path classifier invocations by ~70% in typical workflows
  • After 3 git log invocations, the 4th uses cached strategy (no classify)
  • Warm entries stay in SQLite DB (not in-memory), allowing sharing across processes

Fuzzy Command Matching

NEW in v1.2.2: Command variations share cached strategies via base-command matching.

  • git log --oneline -100 and git log --oneline -50 both use "git log" cache
  • Only 3 total calls to any variant needed to activate warm cache
  • Reduces cold-path invocations by 30-50% in typical workflows

Registry Safety

  • SQLite WAL mode: Multiple concurrent processes can safely read/write without locking conflicts
  • Automatic migration: Legacy registry.json imported on first run
  • Thread safety: Hot cache protected by threading.Lock

🔒 Safety & Security

Automatic Pass-Through Conditions

Clipress passes output through without compression when:

Condition Behavior
Output > 10 MB Pass through (too large to compress safely)
Command in blocklist Pass through (explicitly excluded)
Security pattern matched Pass through (credentials detected)
Binary output detected Pass through (>30% non-ASCII)
Output < 15 lines Pass through (already small)
Compressed > original Return original (size-regression guard)

Built-In Security Patterns

  • Credential files: .env*, *.pem, *.key, id_rsa, id_ed25519
  • Secret keywords: credentials, secret, password, api_key, api-key
  • Token patterns: AWS_SECRET, GITHUB_TOKEN, bearer <token>, -----BEGIN
  • Sensitive commands: printenv, declare, env, set (unconditionally blocked)

Error Handling

Clipress is a compressor, not a validator. On any internal error, it returns the original output unchanged. Set CLIPRESS_DEBUG=1 to surface suppressed errors.

🏗️ Architecture Overview

Module Structure

clipress/
├── engine.py               # Main compression pipeline
├── classifier.py           # Shape detection heuristics
├── learner.py              # SQLite registry + confidence tracking
├── config.py               # Config loading & validation
├── safety.py               # Security gates
├── metrics.py              # Token counting
├── ansi.py                 # ANSI escape code stripping
├── cli.py                  # Click CLI entry point
├── hooks/post_tool_use.py  # Agent hook (Claude Code / Gemini CLI)
└── strategies/
    ├── generic_strategy.py   # Rolling-window deque
    ├── list_strategy.py
    ├── progress_strategy.py
    ├── test_strategy.py
    ├── diff_strategy.py
    ├── table_strategy.py
    ├── keyvalue_strategy.py
    └── error_strategy.py

Compression Flow

cli.compress_cmd() / run_cmd()
  └── engine.compress()
       ├── safety.should_skip()
       ├── config.get_config()
       ├── learner.Learner.lookup()       [SQLite read]
       ├── classifier.detect()             [if not cached]
       ├── strategies[shape].compress()    [rolling-window deque]
       └── learner.Learner.record()       [SQLite write, WAL safe]
